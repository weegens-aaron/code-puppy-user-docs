Understanding Hooks

What Are Hooks?

Hooks are interception points in Code Puppy's execution flow that allow you to inspect, modify, or block operations before they happen. Think of them as security checkpoints or traffic controllers that sit between the agent's decisions and the actual actions.

When an agent wants to do something—run a shell command, read a file, call another tool—hooks give you the opportunity to:

  • Inspect what's about to happen
  • Block dangerous or unwanted operations
  • Modify arguments before execution
  • Log everything for auditing
  • Extend functionality without changing core code

Why Do Hooks Matter?

Hooks are essential for:

Use Case Why It Matters
Safety Prevent accidental rm -rf or destructive operations
Compliance Enforce organizational policies (e.g., no git commands)
Observability Log all agent actions for audit trails
Customization Add your own commands and behaviors
Integration Connect Code Puppy with external tools and workflows

Without hooks, the agent has unrestricted access to your system. With hooks, you maintain control while still benefiting from automation.

How It Works (Simplified)

flowchart LR
    A[Agent decides to run tool] --> B{Hooks Check}
    B -->|Block| C[❌ Operation denied]
    B -->|Allow| D[✅ Tool executes]
    D --> E[Post-hook logs result]

The flow:

  1. Agent requests action — e.g., "run git commit"
  2. Pre-hook fires — Your code/script intercepts the request
  3. Decision point — Allow, block, or modify the request
  4. Action executes (if allowed) — The tool runs
  5. Post-hook fires — Log the result, update metrics

Key Terms

Term Meaning
Pre-hook Fires before an operation (can block)
Post-hook Fires after an operation (observation only)
Matcher Pattern that determines which tools a hook applies to
Callback Python function registered to handle a hook event
Tool An action the agent can perform (run shell, read file, etc.)

Hook Types

Code Puppy supports two complementary hook systems:

1. Script-Based Hooks (Claude Code Compatible)

These work like Claude Code's hooks—shell scripts that intercept tool calls via JSON on stdin.

Best for:

  • Blocking dangerous commands
  • Simple validation logic
  • Integrating with existing shell tools
  • Cross-platform compatibility

Example:

#!/bin/bash
# Block any command with 'rm -rf'
input=$(cat)
command=$(echo "$input" | jq -r '.tool_input.command')

if echo "$command" | grep -q 'rm -rf'; then
  echo "🛑 Blocked: Dangerous command" >&2
  exit 1
fi
exit 0

2. Python Callback Hooks (Code Puppy Native)

These are Python functions registered with Code Puppy's callback system.

Best for:

  • Complex logic and data processing
  • Integration with Python libraries
  • Adding custom commands
  • Real-time streaming events
  • Full access to Code Puppy's internals

Example:

from code_puppy.callbacks import register_callback

def log_tool_call(tool_name: str, tool_args: dict):
    print(f"Tool called: {tool_name} with {tool_args}")

register_callback("pre_tool_call", log_tool_call)

Examples

Example 1: Safety First — Block Destructive Commands

Scenario: You want to prevent the agent from accidentally deleting files.

Script-based hook (.claude/hooks/safety.sh):

#!/bin/bash
input=$(cat)
command=$(echo "$input" | jq -r '.tool_input.command // empty')

# Block destructive patterns
if echo "$command" | grep -qE '(rm -rf|sudo rm|mkfs|dd if=)'; then
  echo "🛑 Safety check: Destructive command blocked" >&2
  echo "Command: $command" >&2
  exit 1
fi
exit 0

Result: When the agent tries rm -rf /tmp/cache, the hook blocks it and shows:

🛑 Safety check: Destructive command blocked
Command: rm -rf /tmp/cache

Example 2: Audit Trail — Log All File Operations

Scenario: You need to track every file the agent reads or modifies.

Python callback hook (~/.code_puppy/plugins/file_audit/register_callbacks.py):

import json
from pathlib import Path
from code_puppy.callbacks import register_callback

AUDIT_LOG = Path.home() / ".code_puppy" / "file_audit.jsonl"

def log_file_operation(tool_name: str, tool_args: dict):
    """Log file read/write operations."""
    if tool_name in ("Read", "Write", "Edit"):
        entry = {
            "tool": tool_name,
            "path": tool_args.get("path"),
            "timestamp": __import__("datetime").datetime.now().isoformat()
        }
        with open(AUDIT_LOG, "a") as f:
            f.write(json.dumps(entry) + "\n")

register_callback("pre_tool_call", log_file_operation)

Result: Every file operation is logged to ~/.code_puppy/file_audit.jsonl:

{"tool": "Read", "path": "/project/src/main.py", "timestamp": "2025-01-15T10:30:00"}
{"tool": "Write", "path": "/project/src/main.py", "timestamp": "2025-01-15T10:31:00"}

Example 3: Custom Command — Add Your Own Slash Command

Scenario: You want a /status command that shows project health.

Python callback hook (~/.code_puppy/plugins/status_cmd/register_callbacks.py):

from code_puppy.callbacks import register_callback
from code_puppy.messaging import emit_info

def handle_status_command(command: str, name: str):
    """Handle /status and related commands."""
    if name == "status":
        emit_info("📊 Project Status:")
        emit_info("  - Files: 42")
        emit_info("  - Tests: Passing ✓")
        emit_info("  - Last commit: 2 hours ago")
        return True  # Indicate we handled this
    return None  # Not handled

register_callback("custom_command", handle_status_command)

Result: When you type /status, you see:

📊 Project Status:
  - Files: 42
  - Tests: Passing ✓
  - Last commit: 2 hours ago

Common Misconceptions

CAUTION

Misconception: Hooks are only for blocking commands Reality: Hooks can also log, modify, extend, and integrate—blocking is just one use case

CAUTION

Misconception: Python callbacks are always better than scripts Reality: Scripts are simpler for basic validation and work across different AI assistants

CAUTION

Misconception: Hooks slow down execution significantly Reality: Well-written hooks add minimal latency (<10ms for most cases)

Tips

TIP

Start simple: Begin with script-based hooks for blocking dangerous commands, then add Python callbacks for complex logic

TIP

Test your hooks: Run them standalone before relying on them: ``bash echo '{"tool_input":{"command":"echo test"}}' | bash .claude/hooks/my-hook.sh ``

TIP

Use meaningful matchers: Instead of "*" (matches everything), be specific: ``json {"matcher": "Bash|agent_run_shell_command", ...} // Only shell commands {"matcher": "Edit && .py", ...} // Only Python file edits ``

TIP

Log failures: When blocking a command, always explain why in stderr so the agent understands: ``bash echo "Blocked: $command violates security policy" >&2 ``
