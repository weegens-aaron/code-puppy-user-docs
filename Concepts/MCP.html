Understanding MCP Integration

What Is MCP?

MCP (Model Context Protocol) is a standard way for AI assistants like Code Puppy to connect with external tools and data sources. Think of it as a universal adapter that lets your AI agent talk to filesystems, databases, browsers, version control systems, and other services.

MCP servers are small programs that expose specific capabilities as "tools" your AI can use. For example:

  • A filesystem server lets your agent read, write, and search files
  • A PostgreSQL server lets your agent query databases
  • A browser server lets your agent interact with web pages

Why Does It Matter?

Without MCP, your AI assistant is limited to what it can do on its own. With MCP, you can:

Capability MCP Server Example What Your Agent Can Do
Work with files filesystem Read source code, create new files, search projects
Query databases postgres, mysql Inspect schemas, run queries, debug data issues
Control browsers browser Navigate websites, take screenshots, extract content
Manage Git git Check commit history, view diffs, manage branches
Access APIs Custom servers Connect to Slack, GitHub, Jira, or any REST API
Run system commands Built-in Execute shell commands (with safety checks)
TIP

MCP is optional — Code Puppy works fine without it. But adding MCP servers dramatically expands what your agent can accomplish.

How It Works (Simplified)

flowchart LR
    A[You ask agent] --> B[Agent receives question]
    B --> C{Does agent have<br/>the right tool?}
    C -->|No| D[Agent asks MCP server<br/>for available tools]
    C -->|Yes| E[Agent uses tool directly]
    D --> F[MCP server exposes<br/>tools like read_file]
    F --> E
    E --> G[Agent executes tool]
    G --> H[You see result]

The Flow in Detail

  1. You ask a question: "What files are in the src/ directory?"
  1. Agent checks capabilities: The agent looks at its available tools. If it doesn't have a list_files tool, it asks connected MCP servers.
  1. MCP server responds: The filesystem MCP server says "I can provide list_files, read_file, write_file, etc."
  1. Agent uses the tool: The agent calls list_files with the path src/.
  1. MCP server executes: The filesystem server runs the actual operation and returns the file list.
  1. Agent shares the result: Your agent tells you what files it found.

Key Terms

Term Meaning
MCP Server A program that exposes tools to AI agents (e.g., filesystem, postgres)
stdio Server runs as a subprocess on your machine (most common)
SSE Server connects over Server-Sent Events (HTTP-based)
Tool A specific capability exposed by an MCP server (e.g., read_file)
Catalog Pre-configured servers you can install with one command
Quarantine Automatic isolation of servers that crash repeatedly

Examples

Example 1: File Operations

Scenario: Your agent needs to review a codebase.

Without MCP: The agent can only see files you explicitly paste into the chat.

With MCP filesystem server:

You: "What's in the src/ directory?"
Agent: [calls list_files tool from filesystem server]
Agent: "I found: main.py, utils.py, config.py"

Example 2: Database Debugging

Scenario: Your app has a data issue.

With MCP PostgreSQL server:

You: "Why are users not logging in?"
Agent: [queries the users table via MCP]
Agent: "I see 3 users with invalid email formats. That's likely the issue."

Common Misconceptions

CAUTION

MCP is only for advanced users: Reality — Installing and using MCP servers is as simple as running /mcp install. You don't need to understand the protocol itself.

CAUTION

MCP servers are always running: Reality — stdio servers start when you run /mcp start <name> and stop when you run /mcp stop <name>. SSE servers are enabled/disabled but don't run as processes.

CAUTION

More servers = better: Reality — Only install servers you actually need. Each server adds complexity and potential failure points. Start with filesystem and add others as required.

Server Types

stdio (Most Common)

Runs as a subprocess on your machine.

Characteristics:

  • Starts/stops with /mcp start / /mcp stop
  • Uses your local system resources
  • Requires the command to be available in your PATH

Example:

{
  "type": "stdio",
  "config": {
    "command": "npx",
    "args": ["-y", "@modelcontextprotocol/server-filesystem"]
  }
}

SSE (Server-Sent Events)

Connects over HTTP with streaming responses.

Characteristics:

  • No subprocess management
  • Connects to remote servers or local services
  • Enabled/disabled but not "started"

Example:

{
  "type": "sse",
  "config": {
    "url": "http://localhost:3000/mcp"
  }
}

Streamable-HTTP

Modern HTTP-based protocol with request/response cycles.

Characteristics:

  • Similar to SSE but uses standard HTTP
  • Better for stateless operations
  • Growing adoption in new servers

Safety & Permissions

Code Puppy includes several safety mechanisms:

Mechanism What It Does
Quarantine Servers that crash 3+ times are automatically disabled
Health monitoring Continuous checks detect unresponsive servers
Permission hooks You can block specific tool calls via hooks
Command validation Shell commands are validated before execution
TIP

To block dangerous tool calls, see Guide: How to block tool calls.

Troubleshooting

Server won't start

Symptoms: /mcp start <name> shows "starting" but never "running"

Common causes:

  1. Missing command: The server's command isn't in your PATH
  2. Wrong arguments: The args configuration is incorrect
  3. Port conflict: For SSE servers, another process uses the port

Fix:

# Check detailed status
/mcp status <server_name>

# Check logs
/mcp logs <server_name>

# Try starting again
/mcp stop <server_name>
/mcp start <server_name>

Agent can't see tools

Symptoms: Server shows "running" but agent doesn't have access to expected tools

Common causes:

  1. Agent not reloaded: After starting a server, the agent needs to reload
  2. Server failed to initialize: Check /mcp status for errors

Fix:

# Reload the agent
/reload

# Or restart Code Puppy entirely
# Then try /mcp start <name> again

Server keeps crashing

Symptoms: Server starts, then immediately stops

Common causes:

  1. Configuration error: Check environment variables or arguments
  2. Resource limits: Server needs more memory or CPU
  3. Dependency missing: Server requires a library or tool not installed

Fix:

# Check the server logs
/mcp logs <server_name>

# Look for error messages in the output
# Fix the configuration, then restart
/mcp restart <server_name>

Tips

TIP

Start with the filesystem server — It's the most useful and works out of the box with npx.

TIP

Use the catalog — /mcp install shows pre-configured servers so you don't have to write JSON by hand.

TIP

Check status regularly — Run /mcp to see which servers are running and healthy.

TIP

Name servers meaningfully — Use descriptive names like my-project-db instead of just postgres so you know what each server does.
