How to Create Custom Skills

What You'll Learn

By the end of this guide, you'll be able to create, structure, and deploy your own custom skills that extend Code Puppy's capabilities with domain-specific knowledge and workflows.

Prerequisites

  • Code Puppy installed and running
  • Basic familiarity with the /skills command (see How to Install Skills)
  • A text editor for creating skill files

Quick Version

# 1. Create skill directory
mkdir -p ~/.code_puppy/skills/my-custom-skill

# 2. Create SKILL.md with frontmatter
cat > ~/.code_puppy/skills/my-custom-skill/SKILL.md << 'EOF'
---
name: my-custom-skill
description: Does something useful
version: 1.0.0
author: Your Name
tags:
  - custom
  - example
---

# My Custom Skill

Detailed instructions go here...
EOF

# 3. Refresh and enable
/skills refresh
/skills enable my-custom-skill

Understanding Custom Skills

Custom skills are your own extensions that teach Code Puppy specialized knowledge. Unlike remote catalog skills, custom skills:

  • Live locally in your skill directories
  • Are fully under your control - edit anytime
  • Can include resources - templates, configs, examples
  • Work offline - no network required
  • Can be shared - version control or share with your team

Basic Skill Structure

A custom skill is a directory with at least one file: SKILL.md.

Minimal Structure

my-custom-skill/
└── SKILL.md    # Required

Recommended Structure

my-custom-skill/
├── SKILL.md              # Required: metadata + instructions
├── templates/            # Optional: reusable templates
│   ├── config.yaml.tmpl
│   └── script.sh.tmpl
├── examples/             # Optional: example files
│   ├── sample.json
│   └── sample.yaml
├── guides/               # Optional: detailed documentation
│   └── best-practices.md
└── scripts/              # Optional: helper scripts
    └── validate.sh

The SKILL.md File

The SKILL.md file has two parts:

  1. YAML Frontmatter - Metadata for discovery and identification
  2. Markdown Content - Instructions for the agent

YAML Frontmatter

---
name: my-custom-skill
description: Brief description of what the skill does
version: 1.0.0
author: Your Name <email@example.com>
tags:
  - tag1
  - tag2
  - tag3
---

Required Fields

Field Type Description
name string Unique identifier (kebab-case recommended)
description string One-sentence summary of the skill's purpose

Optional Fields

Field Type Description
version string Semantic version (e.g., "1.0.0")
author string Author name or contact
tags list Keywords for categorization and search

Markdown Content

After the frontmatter, write detailed instructions in Markdown:

# Skill Name

## When to Use This Skill

Describe scenarios where this skill applies.

## Instructions

Step-by-step guidance for the agent.

## Best Practices

Recommendations and guidelines.

## Examples

Concrete examples of usage.

## Resources

List any bundled resources and how to use them.

Example: Creating a Git Helper Skill

Let's create a practical skill for common Git operations.

Step 1: Create the Directory

mkdir -p ~/.code_puppy/skills/git-helper

Step 2: Create SKILL.md

cat > ~/.code_puppy/skills/git-helper/SKILL.md << 'EOF'
---
name: git-helper
description: Common Git operations including status, branching, commits, and history
version: 1.0.0
author: You
tags:
  - git
  - version-control
  - scm
---

# Git Helper Skill

## When to Use This Skill

Use this skill when the user needs help with:
- Checking Git repository status
- Creating and managing branches
- Writing commit messages
- Viewing commit history
- Understanding Git workflows

## Instructions

### 1. Check Repository Status

Always start by understanding the current state:

git status --short git diff --stat


### 2. Branch Management

For new features:
1. Create a feature branch from main
2. Use descriptive branch names (e.g., `feature/add-auth`)
3. Keep branches focused and small

### 3. Commit Best Practices

- Write clear, concise commit messages
- Use the imperative mood ("Add feature" not "Added feature")
- Keep commits atomic (one logical change per commit)
- Include relevant context in the commit body

### 4. Common Workflows

#### Feature Development

Create and switch to feature branch

git checkout -b feature/feature-name

Make changes and commit

git add . git commit -m "feat: add feature description"

Push and create PR

git push -u origin feature/feature-name


#### Bug Fixes

Create hotfix branch

git checkout -b hotfix/bug-description

Fix and commit

git add . git commit -m "fix: describe the bug fix"

Push and create PR

git push -u origin hotfix/bug-description


## Tips

> [!TIP]
> Always review `git status` before making changes to understand the current state.

> [!TIP]
> Use `git diff` to review changes before committing.

> [!TIP]
> Keep commit messages under 72 characters for the subject line.

## Resources

This skill works with standard Git commands. No additional resources are required.
EOF

Step 3: Verify the Skill

# Check the file exists
ls -la ~/.code_puppy/skills/git-helper/SKILL.md

# Verify frontmatter parsing
head -20 ~/.code_puppy/skills/git-helper/SKILL.md

Example: Creating a Skill with Resources

Skills can include templates, configs, and examples.

Step 1: Create Directory Structure

mkdir -p ~/.code_puppy/skills/docker-expert/{templates,examples}

Step 2: Create SKILL.md

---
name: docker-expert
description: Docker containerization, Dockerfile optimization, and docker-compose orchestration
version: 1.0.0
author: You
tags:
  - docker
  - containers
  - devops
---

# Docker Expert Skill

## When to Use This Skill

Use this skill when the user needs help with:
- Writing or optimizing Dockerfiles
- Setting up docker-compose configurations
- Container best practices and security
- Multi-stage builds for smaller images

## Instructions

### 1. Dockerfile Best Practices

Always follow these principles:
- Use specific base image tags (not `latest`)
- Leverage multi-stage builds to minimize image size
- Combine RUN commands to reduce layers
- Use `.dockerignore` to exclude unnecessary files

### 2. Security Guidelines

- Run containers as non-root users
- Scan images for vulnerabilities
- Minimize installed packages
- Use distroless or slim base images when possible

### 3. Available Templates

This skill includes the following templates:
- `templates/Dockerfile.multi-stage` - Multi-stage build template
- `templates/docker-compose.yml` - Basic compose template
- `templates/.dockerignore` - Common ignore patterns

Use these as starting points and customize as needed.

## Resources

- `templates/Dockerfile.multi-stage` - Multi-stage Dockerfile template
- `templates/docker-compose.yml` - Docker Compose template
- `templates/.dockerignore` - Common ignore patterns
- `examples/node-app/Dockerfile` - Example Node.js Dockerfile
- `examples/python-app/Dockerfile` - Example Python Dockerfile
EOF

Step 3: Add Template Files

templates/Dockerfile.multi-stage:

# Multi-stage build template
# Replace <base-image> and <workdir> as needed

# Stage 1: Builder
FROM <base-image>:<version>-alpine AS builder
WORKDIR <workdir>

# Copy dependency files first (better caching)
COPY package*.json ./
RUN npm ci --only=production

# Copy source and build
COPY . .
RUN npm run build

# Stage 2: Runtime
FROM <base-image>:<version>-alpine

# Create non-root user
RUN addgroup -g 1000 app && \
    adduser -u 1000 -G app -s /bin/sh -D app

WORKDIR <workdir>

# Copy built artifacts from builder
COPY --from=builder <workdir>/dist ./dist
COPY --from=builder <workdir>/node_modules ./node_modules

# Run as non-root
USER app

CMD ["node", "dist/index.js"]

templates/docker-compose.yml:

version: '3.8'

services:
  app:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
    restart: unless-stopped
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 10s
      retries: 3

  # Optional: Add services as needed
  # redis:
  #   image: redis:7-alpine
  #   ports:
  #     - "6379:6379"

Step 4: Add Example Files

examples/node-app/Dockerfile:

FROM node:18-alpine AS builder
WORKDIR /app

COPY package*.json ./
RUN npm ci --only=production

COPY . .
RUN npm run build

FROM node:18-alpine
WORKDIR /app

COPY --from=builder /app/dist ./dist

Testing Your Custom Skill

Step 1: Refresh the Skill Cache

/skills refresh

You should see:

✓ Skill cache refreshed
Found 1 new skill(s)

Step 2: List Your Skills

/skills list

Your skill should appear:

🛠️ Skills (integration: enabled)
Found 5 skills:

  🟢 enabled docker-expert v1.0.0 by You
      Docker containerization, Dockerfile optimization...
      tags: docker, containers, devops

  🟢 enabled git-helper v1.0.0 by You
      Common Git operations including status, branching...
      tags: git, version-control, scm

Step 3: Enable the Skill

If not automatically enabled:

/skills enable git-helper

Step 4: Test Activation

Start a conversation with an agent and ask a relevant question:

/agent code-puppy

# Then ask:
Help me create a Dockerfile for my Node.js application

The agent should:

  1. Detect the relevant skill from <available_skills> in context
  2. Call activate_skill(skill_name="docker-expert")
  3. Follow the instructions from your SKILL.md

Advanced: Skill with Scripts

Skills can include executable scripts that agents can reference or run.

Structure

script-helper/
├── SKILL.md
└── scripts/
    ├── validate-config.sh
    └── backup-database.sh

SKILL.md Excerpt

---
name: script-helper
description: Helper scripts for common operations
version: 1.0.0
author: You
tags:
  - scripts
  - automation
---

# Script Helper Skill

## Available Scripts

### validate-config.sh

Validates configuration files (YAML, JSON, TOML).

**Usage:**

./scripts/validate-config.sh path/to/config.yaml


**Output:**
- Exit code 0: Valid
- Exit code 1: Invalid with error details

### backup-database.sh

Creates database backups with timestamp.

**Usage:**

./scripts/backup-database.sh --database mydb --output /backups


**Output:**
- Backup file: `mydb-YYYYMMDD-HHMMSS.sql.gz`
- Exit code 0: Success
- Exit code 1: Failure

Best Practices

Naming Conventions

  • Use kebab-case for skill names (my-skill, not mySkill or my_skill)
  • Keep names descriptive but concise
  • Avoid generic names like utils or helpers
  • Prefix domain-specific skills (aws-s3, gcp-cloudrun)

Writing Instructions

  • Be specific - "Use Python 3.11+" not "Use Python"
  • Include examples - Show, don't just tell
  • Cover edge cases - What if something goes wrong?
  • Version when relevant - "As of 2024" or "For Node.js 18+"

Organizing Resources

  • Group related files in subdirectories
  • Use clear, descriptive filenames
  • Include a README in complex skills
  • Keep templates generic but customizable

Versioning

  • Use semantic versioning (major.minor.patch)
  • Increment version when:
  • Adding new features (minor)
  • Making breaking changes (major)
  • Fixing bugs (patch)

Troubleshooting

Problem Cause Solution
Skill not appearing Missing SKILL.md Add valid SKILL.md to skill directory
Skill shows "no metadata" Invalid frontmatter Check YAML syntax, ensure name and description are present
Skill not activating Skill disabled Run /skills enable <skill-name>
Instructions not loading File encoding issue Ensure SKILL.md is UTF-8 encoded
Resources not found Wrong path Use relative paths from skill directory

Tips & Tricks

TIP

Share Your Skills: Put custom skills in a Git repo and share with your team. Everyone can clone to their ~/.code_puppy/skills/ directory.

TIP

Version Control: Track your custom skills in Git. This lets you: - Share with team members - Roll back changes - Maintain a history of improvements

TIP

Skill Templates: Create a "starter skill" template with proper structure. Clone it when creating new skills.

TIP

Test Iteratively: Make small changes and test frequently. Use /skills refresh after each change.

TIP

Document Dependencies: If your skill requires external tools or services, document them clearly in the SKILL.md.
