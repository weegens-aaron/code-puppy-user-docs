Tutorial: Setting Up Multi-Model Distribution

What You'll Achieve

By the end of this tutorial, you'll have configured a round-robin model that distributes requests across multiple models, helping you overcome rate limits and balance load across different API keys or providers.

Before You Begin

Time needed: ~10 minutes

You'll need:

  • Code Puppy installed and running
  • At least 2 models configured (or API keys for the same model type)
  • Basic familiarity with the Quick Start

You should already know:

  • How to run Code Puppy
  • How to use the /model command to switch models

The Scenario

You're working on a large coding project that requires many AI interactions. You've hit rate limits with your single API key, or you want to:

  • Maximize throughput by distributing requests across multiple API keys
  • Balance costs by mixing expensive and cheaper models
  • Increase reliability by having fallback options
  • Test multiple models to see which works best for your use case

Multi-model distribution solves this by creating a virtual model that cycles through your available models automatically.

Step 1: Choose Your Distribution Strategy

Decide what you want to achieve:

Goal Strategy
Avoid rate limits Multiple keys for the same model
Cost optimization Mix expensive + cheaper models
Model comparison Cycle through different model types
High availability Redundant models as fallbacks
TIP

For this tutorial, we'll set up multiple Cerebras API keys to maximize throughput. Adapt the steps for your use case.

Step 2: Prepare Your Base Models

You need individual models configured before creating the round-robin. Let's set up three Cerebras keys:

**Create or edit your ~/.code_puppy/models.json:**

{
  "cerebras_key1": {
    "type": "cerebras",
    "name": "zai-glm-4.7",
    "custom_endpoint": {
      "url": "https://api.cerebras.ai/v1",
      "api_key": "$CEREBRAS_API_KEY1"
    },
    "context_length": 131072,
    "supported_settings": ["temperature", "seed", "top_p"]
  },
  "cerebras_key2": {
    "type": "cerebras",
    "name": "zai-glm-4.7",
    "custom_endpoint": {
      "url": "https://api.cerebras.ai/v1",
      "api_key": "$CEREBRAS_API_KEY2"
    },
    "context_length": 131072,
    "supported_settings": ["temperature", "seed", "top_p"]
  },
  "cerebras_key3": {
    "type": "cerebras",
    "name": "zai-glm-4.7",
    "custom_endpoint": {
      "url": "https://api.cerebras.ai/v1",
      "api_key": "$CEREBRAS_API_KEY3"
    },
    "context_length": 131072,
    "supported_settings": ["temperature", "seed", "top_p"]
  }
}

Set your environment variables:

export CEREBRAS_API_KEY1="your-first-api-key-here"
export CEREBRAS_API_KEY2="your-second-api-key-here"
export CEREBRAS_API_KEY3="your-third-api-key-here"
NOTE

Each model name (cerebras_key1, cerebras_key2, etc.) must be unique. The name field inside each config can be the same if you're using the same model type.

Step 3: Create the Round-Robin Configuration

Now add a round-robin model that references your base models:

**Add to the same models.json file:**

{
  "cerebras_key1": { ... },
  "cerebras_key2": { ... },
  "cerebras_key3": { ... },
  "cerebras_round_robin": {
    "type": "round_robin",
    "models": ["cerebras_key1", "cerebras_key2", "cerebras_key3"],
    "rotate_every": 1
  }
}

Configuration options:

Option Description Example
type Must be "round_robin" "round_robin"
models List of model names to cycle through ["model1", "model2", "model3"]
rotate_every Requests per model before rotating (default: 1) 1, 5, 10

**Understanding rotate_every:**

  • **rotate_every: 1** (default): Switch after every request ``` Request 1 → Key 1 Request 2 → Key 2 Request 3 → Key 3 Request 4 → Key 1 (cycle repeats) ```
  • **rotate_every: 5**: Stay on each model for 5 requests ``` Requests 1-5 → Key 1 Requests 6-10 → Key 2 Requests 11-15 → Key 3 Requests 16-20 → Key 1 (cycle repeats) ```
TIP

Use rotate_every: 1 for maximum distribution. Use higher values if you want to reduce connection overhead or keep sessions on the same model longer.

Step 4: Reload Your Configuration

If Code Puppy is already running, reload the models:

/models reload

Or restart Code Puppy to pick up the new configuration:

code-puppy

Step 5: Select the Round-Robin Model

Use the model picker to activate your round-robin configuration:

/model

Tab through the list until you see cerebras_round_robin and press Enter.

You should see:

round_robin:cerebras_key1,cerebras_key2,cerebras_key3

This confirms you're using the round-robin distribution.

Step 6: Verify Distribution is Working

Make several requests and watch the model name change:

Request 1:

> Please analyze this code for bugs...

Model shows: round_robin:... (using Key 1)

Request 2:

> Now refactor this function...

Model shows: round_robin:... (using Key 2)

Request 3:

> Write tests for this module...

Model shows: round_robin:... (using Key 3)

Request 4:

> Optimize this algorithm...

Model shows: round_robin:... (using Key 1 again)

NOTE

The underlying key rotates automatically. You don't need to manually switch models.

Final Result

You now have a round-robin model that:

  • ✅ Distributes requests across 3 API keys
  • ✅ Effectively triples your rate limit (300 req/hr instead of 100)
  • ✅ Works transparently — just use it like any other model
  • ✅ Automatically cycles through all configured models

Your effective rate limit:

3 keys × 100 requests/hour = 300 requests/hour

What You Learned

  • Round-robin models distribute requests across multiple models
  • Configuration requires base models + a round-robin wrapper
  • **rotate_every** controls how often to switch models
  • Environment variables can be referenced with $VAR_NAME syntax
  • Distribution happens automatically during normal use

Variations to Try

Cost Optimization Mix

{
  "expensive_model": {
    "type": "anthropic",
    "name": "claude-opus-4-5",
    ...
  },
  "cheap_model": {
    "type": "cerebras",
    "name": "zai-glm-4.7",
    ...
  },
  "cost_balanced": {
    "type": "round_robin",
    "models": ["expensive_model", "cheap_model"],
    "rotate_every": 2
  }
}

This uses the expensive model for 2 requests, then the cheap model for 2 requests, balancing quality and cost.

Provider Redundancy

{
  "openai_fallback": {
    "type": "round_robin",
    "models": ["gpt-5.2", "claude-4-5-sonnet", "Gemini-3"],
    "rotate_every": 1
  }
}

Distributes across providers for maximum reliability and to compare model performance.

Troubleshooting

Problem Solution
ValueError: Round-robin model requires a 'models' list Ensure models is a non-empty array
Model not found: cerebras_key1 Verify all referenced models exist in your config
rotate_every must be at least 1 Set rotate_every to 1 or higher
Not seeing rotation Make more requests — rotation happens per request
Environment variable not found Export the variable or use a literal API key
